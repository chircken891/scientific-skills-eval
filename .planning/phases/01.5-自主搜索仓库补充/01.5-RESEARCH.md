Validation criteria include star count, recent commits, active maintenance, and quality signals. Deduplication against the existing 21 repos is mandatory per D-04/D-05.

**Primary recommendation:** Use GitHub API with `User-Agent` header for systematic search, combined with npm registry search for MCP servers and academic tools. Focus search on compound queries combining research domains with "claude-skill" or "mcp" keywords.

## Common Pitfalls

### Pitfall 1: GitHub API rate limiting without authentication
**What goes wrong:** Unauthenticated requests limited to 10/hour, search fails
**Why it happens:** No authentication token provided
**How to avoid:** Use `gh auth login` or provide `-H "Accept: application/vnd.github.v3+json"` with User-Agent
**Warning signs:** 403 Forbidden response with "Resource not found" or "Rate limit exceeded"

### Pitfall 2: Missing User-Agent header
**What goes wrong:** GitHub API returns 403 with "User-Agent header required"
**Why it happens:** GitHub requires User-Agent on all requests
**How to avoid:** Always include `-H "User-Agent: research-skill-evaluator"` or similar
**Warning signs:** JSON error response with "Request forbidden by administrative rules"

### Pitfall 3: Confusing npm package name with repository name
**What goes wrong:** npm package "grd-cli" maps to GitHub repo with different name
**Why it happens:** npm and GitHub namespaces are independent
**How to avoid:** Always verify the npm package points to a valid, accessible repository
**Warning signs:** Package has GitHub URL in homepage field that doesn't match expected repo

### Pitfall 4: Ignoring MCP servers as separate category
**What goes wrong:** MCP servers (researchmcp, openalex-research-mcp) are not traditional "skills" but provide skill-like functionality
**Why it happens:** MCP is a protocol, not a skill format
**How to avoid:** Include MCP servers as "related tools" in SKILLS-INVENTORY, even if not traditional skills
**Warning signs:** Package has `model-context-protocol` in keywords

### Pitfall 5: Not checking existing 21 repos before adding
**What goes wrong:** Duplicating repositories already in Phase 1 results
**Why it happens:** Cross-platform search finds same repo from different angles
**How to avoid:** Deduplicate against existing SKILLS-INVENTORY.md entries using `owner/name` as key
**Warning signs:** Repo already appears in existing inventory with same URL

## Code Examples

### GitHub Search with User-Agent (verified)
```bash
curl -s "https://api.github.com/search/repositories?q=claude+skill+scientific+research&per_page=20&sort=stars" \
  -H "User-Agent: research-skill-evaluator" \
  -H "Accept: application/vnd.github.v3+json"
```

### npm Search for Academic Research
```bash
npm search "academic research claude" 2>/dev/null | head -30
```

### Extract Key Fields from GitHub Response
```bash
curl -s "https://api.github.com/search/repositories?q=paper+writing+claude+skill&per_page=20" \
  -H "User-Agent: research-skill-evaluator" | \
  jq '.items[] | {name, full_name, stars: .stargazers_count, url: .html_url, topics: .topics}'
```
