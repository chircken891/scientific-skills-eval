# Phase 01.5.1: 评测方法预测试 — Research

**Researched:** 2026-05-11
**Domain:** Methodology design for skill evaluation scoring system
**Confidence:** HIGH

## Summary

This is a methodology-validation phase, not a software engineering phase. The goal is to redesign the skill evaluation scoring system from a "generalist" framework (which penalizes specialized skills) to a "specialist depth" framework, then pretest it on 10 samples to validate the methodology works before Phase 2 full execution.

The core architectural change: replace the old 40/30/30 weighted sum model with a two-tier evaluation. **Tier 1: Domain Quality Rating (1-5)** — scores only the 1-2 functions each skill does best, determines exclusion/candidate/recommend. **Tier 2: Descriptive Supplementary Info** — coverage of research phases and integration characteristics, informs combination decisions but does not factor into the rating.

**Primary recommendation:** The phase should produce three concrete deliverables: (1) new scoring rubric definition, (2) re-scored 10 samples with new system, (3) validated methodology specification for Phase 2. Organize into 2-3 plans with clear separation between methodology design and execution.

## Current vs Proposed Scoring Architecture

### Old System (Baseline)

```
[4 security checks: pass/fail] → veto independent
        │
[Functional 40%: 8 subdimensions x 5] = 40 pts max
[Integration 30%: 5 subdimensions x 5] = 30 pts max  →  Weighted Sum (100 pts)
[Coverage 30%: 4 subdimensions x 5] = 30 pts max    /
```

**Problem:** A literature-review skill with 8 functional dimensions — 4 dimensions score low (stats, code gen, multimodal, etc.) because they're irrelevant to literature review. Total: ~20/40 penalty for being specialized.

### New System (Proposed)

```
[4 security checks: pass/fail] → independent veto
                                      ┌──────────────────────┐
[Functional depth: score only         │ TIER 1 (1-5 rating)  │
 the 1-2 best functions]  ───────────→│ = decision driver    │
                                      └──────────────────────┘
                                                    │
                                          ┌─────────┴──────────┐
                                          ▼                    ▼
                                     < 3: EXCLUDE    3-4: CANDIDATE   >4: AUTO-RECOMMEND
                                          │                    │              │
                                          ▼                    ▼              ▼
                                    [reason required]   [auto-classify]  [auto-classify]
                                                    │
                    ┌───────────────────────────────┘
                    ▼
[Coverage: descriptive] ──────→┐
[Integration: descriptive] ────→├── TIER 2 (combination input, not scored)
                                │    Used for: which skills combine well
                                │    Human confirms final combination
```

**Key insight:** Tier 1 (1-5 rating) drives the go/no-go decision. Tier 2 (qualitative) drives the how-to-combine decision. They are independent.

## Architecture Patterns

### Pattern 1: Two-Tier Evaluation Separation

**What:** Separate the "quality rating" (Tier 1) from the "compatibility info" (Tier 2). Tier 1 drives exclusion/candidate/recommend decisions. Tier 2 informs combination decisions.

**Why:** Prevents cross-contamination between "is this skill good at what it does" and "does this skill fit with others." A highly specialized skill might be 5/5 in its domain (Tier 1) but cover only the "analysis" phase (Tier 2) — human decides whether to combine it with other skills.

**Decision flow:**
```
         Security
            │
            ▼ (PASS only)
  Professional Depth Score (1-5)
            │
     ┌──────┼──────┐
     ▼      ▼      ▼
   <3     3-4    >4
  EXCLUDE CANDIDATE AUTO-RECOMMEND
     │      │        │
     ▼      ▼        ▼
     All enter Tier 2 analysis (excluded items flagged for human review)
     Coverage + Integration info → combination recommendations
                               → HUMAN confirms final portfolio
```

### Pattern 2: Professional Depth Rubric Per Function Type

**What:** Define what 1-5 means for each major function type. The same numeric score has different qualitative meanings depending on what function is being evaluated.

**Why:** "5/5" for literature search means something different from "5/5" for data analysis. A function-agnostic rubric would miss nuance.

**Rubric structure example (not exhaustive, needs definition in planning):**

| Score | Literature Search | Data Analysis | Writing | Figure Generation |
|-------|-------------------|---------------|---------|-------------------|
| 5/5 | Multi-database, systematic review, quality filtering | Full statistical workflow, hypothesis testing, modeling | Journal-adaptive style, multi-round revision, review simulation | Publication-ready, multi-format, style transfer from papers |
| 4/5 | Broad database coverage, structured search | Common analyses, visualization, reporting | Structured writing, templates, basic revision | Good quality, common formats, manual tuning |
| 3/5 | Single database, basic search | Descriptive stats, simple plots | Coherent prose, basic structure | Competent but generic |
| 2/5 | Limited search capability | Minimal analysis features | Produces text but structurally weak | Basic output, needs heavy editing |
| 1/5 | Vague search guidance | No real analysis capability | Output is outline-level at best | Cannot generate usable figures |

### Anti-Patterns to Avoid

- **Averaging irrelevant dimensions:** Under the OLD system, scoring a literature-review skill on "code generation" or "multimodal" makes no sense but drags the score down. The new system MUST explicitly exclude irrelevant dimensions. [CITED: D-06 principle]
- **Mixing quality and compatibility in one number:** A skill could be excellent (5/5) but redundant with another skill. These are different decisions. Keep them separate. [ASSUMED: informed by two-tier structure]
- **Over-defining the rubric:** The rubric should be specific enough to ensure consistency but not so detailed that evaluating each skill becomes mechanical busywork. Target: 2-3 diagnostic questions per function type, not a 10-item checklist. [ASSUMED]

### Pattern 3: Pipeline Scoring Approach (for academic-pipeline)

**What:** academic-pipeline is a 10-stage chained pipeline, not a single skill. Under the professional depth model, its "function" is workflow orchestration.

**Scoring approach:**
- **Core function:** Workflow orchestration — how well does it chain stages, handle state, provide checkpoints?
- **Secondary function:** Research stage coverage — how many stages does it automate (literature → analysis → writing → submission)?
- **What NOT to score:** It's not a data analysis skill, so don't score it on statistical capability.
- **Tier 2 description:** Describe which stages it covers and what integration interfaces it provides.

## Common Pitfalls

### Pitfall 1: Function Selection Subjectivity
**What goes wrong:** Two evaluators looking at the same skill might pick different "best 1-2 functions," leading to inconsistent ratings.
**Why it happens:** The scoring guide doesn't specify criteria for choosing which functions to score.
**How to avoid:** Define a priority rule for function selection. For example: "If the skill has a self-declared specialization (stated in its description), score that first. If it claims multiple specializations, score the one with the strongest evidence of quality."
**Warning signs:** If post-reevaluation of the 9 samples shows ratings that are inconsistent with common sense (e.g., medsci-skills getting less than 4/5), check function selection criteria.

### Pitfall 2: Pipeline Skew
**What goes wrong:** academic-pipeline (10-stage pipeline) gets unfairly compared to single-purpose skills because "workflow orchestration" is hard to assess on a 1-5 scale.
**Why it happens:** The rubric was designed for individual skills, not pipelines.
**How to avoid:** Recognize that pipeline scoring is a separate rubric category. Score it on: (1) stage reliability — do all 10 stages execute correctly? (2) checkpoint quality — can you intervene at each stage? (3) stage integration — do outputs flow cleanly between stages?
**Warning signs:** If academic-pipeline gets an extreme score (< 2 or 5/5), the rubric may not fit pipelines.

### Pitfall 3: Granularity Loss from 100-point to 5-point Scale
**What goes wrong:** With 100-point scale, a 5-point difference is meaningful resolution. With 1-5 scale, the difference between 3 and 4 is a large information loss.
**Why it happens:** The 1-5 scale has only 5 levels vs the old 100-point scale.
**How to avoid:** Use 0.5 increments (3.0, 3.5, 4.0, 4.5) within the 1-5 range. This gives 9 levels of differentiation while keeping the categorical thresholds clear (<3, 3-4, >4). This is a recommendation for the planning phase to decide.
**Warning signs:** If most skills cluster at 3-4 (all look the same), the scale is too coarse.

### Pitfall 4: Threshold Boundary Ambiguity
**What goes wrong:** A skill that scores exactly 3.0 or 4.0 falls at a boundary. Is 3.0 "exclude" or "candidate"? Is 4.0 "candidate" or "auto-recommend"?
**Why it happens:** The boundary conditions are not defined in the current decisions (D-07/D-08/D-09).
**How to avoid:** Define inclusive/exclusive boundaries. Recommended: 3.0 = candidate (lower bound included in candidate), 4.0 = auto-recommend (upper bound included in auto-recommend). Explicitly decide during planning.
**Warning signs:** Boundary skills (< 5% of sample) at exactly 3.0 or 4.0 need explicit rules.
