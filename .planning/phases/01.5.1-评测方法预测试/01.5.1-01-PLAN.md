## Scoring Rules (from Research)

1. **Half-point increments:** Use 0.5 granularity within 1-5 (1.0, 1.5, 2.0, ..., 5.0) giving 9 levels of differentiation
2. **Boundary definitions:** < 3.0 = exclude, 3.0-4.0 = candidate (lower bound inclusive), > 4.0 = auto-recommend (upper bound inclusive)
3. **Depth score aggregation:** Average of the 1-2 selected function scores = DepthScore
4. **Function selection priority:** If skill has self-declared specialization, score that first. If multiple specializations, score the one with strongest evidence of quality

## Section 1: Scoring Philosophy (D-06)
State the core principle: "This rubric evaluates how well a skill performs in its 1-2 strongest functions. Specialized skills are NOT penalized for lacking capabilities outside their stated focus. The question is 'how good at what it does best' not 'how many things can it do.'"

## Section 2: Scoring Rules
Document these exact rules:
- Scale: 1.0 to 5.0, 0.5 increments (9 levels: 1.0, 1.5, 2.0, 2.5, 3.0, 3.5, 4.0, 4.5, 5.0)
- Depth score = average of 1-2 selected function scores (round to nearest 0.5)
- Function Selection Priority Rule (from RESEARCH.md Pitfall 1):
  1. If skill has self-declared specialization in its README/description → score that first
  2. If multiple specializations claimed → score the one with strongest code/documentation evidence of quality
  3. If no clear specialization → score the 2 functions where the skill provides the most concrete implementation detail
- Pipeline scoring rule (from RESEARCH.md Pattern 3):
  - For pipeline/orchestrator skills (like academic-pipeline), the core function is "workflow orchestration"
  - Score on: stage reliability (do stages execute correctly), checkpoint quality (can user intervene at each stage), stage integration (do outputs flow cleanly between stages)
  - Do NOT score pipeline on individual stage capabilities (e.g., don't score a pipeline on "literature search quality" if it orchestrates literature search stages)
