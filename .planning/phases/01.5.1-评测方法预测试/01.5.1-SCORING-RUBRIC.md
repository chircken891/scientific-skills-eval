**Version:** 1.0
**Phase:** 01.5.1 (评测方法预测试)
**Status:** Draft for pretest validation

## Section 1: Scoring Philosophy (D-06)

This rubric evaluates how well a skill performs in its 1-2 strongest functions. Specialized skills are NOT penalized for lacking capabilities outside their stated focus. The question is "how good at what it does best" not "how many things can it do."

### Core Principles

1. **Depth over breadth:** A skill that does one thing excellently scores higher than a skill that does many things adequately.
2. **Specialization is a feature, not a bug:** Skills explicitly designed for narrow domains (e.g., citation management, Nature journal formatting) are evaluated on their stated purpose, not on general-purpose capability.
3. **Evidence-based scoring:** Scores are grounded in observable artifacts — code quality, documentation clarity, demonstrated workflows. Subjective "this feels good" has no place.
4. **Function selection first:** The rater MUST select the 1-2 best functions BEFORE assigning scores. Score reflects the selected functions, not the entire skill's feature list.

## Section 2: Scoring Rules

### Scale Definition

| Property | Value |
|----------|-------|
| Range | 1.0 to 5.0 |
| Increment | 0.5 |
| Total levels | 9 (1.0, 1.5, 2.0, 2.5, 3.0, 3.5, 4.0, 4.5, 5.0) |

### Depth Score Calculation

```
DepthScore = average(Function1_Score, Function2_Score)
```

- If only 1 function is scored: `DepthScore = Function1_Score`
- If 2 functions are scored: `DepthScore = round_to_nearest_0.5((F1 + F2) / 2)`
- Rounding: 0.25 rounds up, 0.24 rounds down (e.g., 3.25 → 3.5, 3.24 → 3.0)

### Function Selection Priority Rule

When selecting which 1-2 functions to score, apply these rules in order:

1. **Self-declared specialization:** If the skill's README, description, or title explicitly states its specialization (e.g., "A citation management skill for researchers"), score that function first.
2. **Strongest evidence:** If multiple specializations are claimed, score the one with the strongest code/documentation evidence of quality — look for concrete implementations, test coverage, detailed documentation, and working examples.
3. **Concrete implementation:** If no clear specialization is stated, score the 2 functions where the skill provides the most concrete implementation detail (actual code, detailed prompts, workflow specifications).

### Pipeline / Orchestrator Scoring Rule

For pipeline or orchestrator skills (where the skill chains together multiple stages or sub-skills):

- **Core function is "workflow orchestration."** Score on:
  - Stage reliability: Do individual stages execute correctly and consistently?
  - Checkpoint quality: Can the user review, intervene, or modify at each stage?
  - Stage integration: Do outputs flow cleanly between stages without data loss or corruption?
  - Error handling: What happens when a stage fails? Is there recovery or graceful degradation?
- **DO NOT** score the pipeline on individual stage capabilities. For example, if a pipeline orchestrates literature search stages, do NOT score its "literature search quality" — that's the responsibility of the underlying search skill. Score only the orchestration itself.

## Section 3: Threshold Mapping (D-07 / D-08 / D-09)

| Depth Score | Classification | Action |
|-------------|----------------|--------|
| < 3.0 | EXCLUDE | Must provide exclusion reason per D-10 |
| 3.0 - 4.0 | CANDIDATE | Auto-classified as candidate per D-08 |
| > 4.0 | AUTO-RECOMMEND | Auto-selected for portfolio recommendation per D-09 |

### Boundary Rules

- **3.0** is CANDIDATE (lower bound inclusive) — a skill scoring exactly 3.0 is a candidate, not excluded.
- **4.0** is AUTO-RECOMMEND (upper bound inclusive) — a skill scoring exactly 4.0 is auto-recommended, not a candidate.
- Classification is determined by DepthScore only after security check. A SecurityVeto overrides the threshold (→ immediate EXCLUDE).

## Section 4: Rubric Table Per Function Type

### 4.1 Literature Search

For skills that search academic databases, formulate systematic queries, filter by quality, and deduplicate results. Representative repo: ARIS.

| Score | Criteria |
|-------|----------|
| 5/5 | Multi-database search (PubMed, arXiv, Semantic Scholar, Crossref, etc.) with systematic query formulation (PICO, Boolean operators, MeSH terms). Automated quality filtering by venue rank/citation count. Built-in deduplication across sources. Results export with full metadata (BibTeX, RIS). Search strategy documented in machine-readable format. |
| 4/5 | Supports 3+ databases with Boolean query support. Basic quality filtering (venue rank only). Deduplication available but may miss edge cases. Results export in 1-2 formats. Search strategy partially documented. |
| 3/5 | Supports 1-2 databases with keyword search only. No systematic query tools. Manual filtering required. Basic deduplication (identical duplicate removal only). Results export in single format. |
| 2/5 | Single-database search with limited query syntax. No quality filtering. No deduplication. Output is raw JSON/text without structured metadata. |
| 1/5 | Minimal or no literature search capability. May provide general web search with no academic focus. No structured output. |

### 4.2 Literature Review / Systematic Review

For skills that perform structured review workflows, synthesize findings across sources, and produce formatted reviews. Representative repo: AI-Powered-Literature-Review-Skills.

| Score | Criteria |
|-------|----------|
| 5/5 | Full systematic review workflow: PRISMA-compliant screening, abstract/full-text filtering, risk of bias assessment, evidence table generation. Database integration with PubMed Central, Cochrane, etc. Thematic synthesis with citation network analysis. Auto-generates PRISMA flow diagram. Exports to structured review formats. |
| 4/5 | Structured review with title/abstract screening and data extraction. Supports inclusion/exclusion criteria. Basic quality assessment. Thematic grouping of findings. Exports summary tables. |
| 3/5 | Literature summary with key finding extraction. Basic categorization by theme. Can answer specific review questions. No systematic methodology adherence. |
| 2/5 | Simple literature list with one-sentence summaries per paper. No extraction, no quality assessment, no synthesis. |
| 1/5 | Cannot perform structured literature review. May list papers without meaningful synthesis or analysis. |

### 4.9 Workflow Orchestration / Pipeline

For skills that chain multiple stages into a complete workflow, with checkpoints, data flow management, and error handling. Representative repo: academic-pipeline.

| Score | Criteria |
|-------|----------|
| 5/5 | Full multi-stage pipeline (8+ stages) covering complete research workflow. Each stage has documented inputs/outputs. Checkpoint at every stage allowing user review, modification, and resumption. Cross-stage data validation (output schema checks between stages). Error handling with recovery options (retry, skip, replace stage). Pipeline state persistence across sessions. Parallel stage execution where independent. |
| 4/5 | Multi-stage pipeline (5-7 stages). Checkpoints at major stages. Basic data validation between stages. Error handling with graceful failure. State persistence within session. |
| 3/5 | Linear pipeline (3-4 stages) with sequential execution. Single checkpoint option. Minimal error handling (fails on any error). Basic stage-to-stage data passing. |
| 2/5 | 2-stage chaining. No checkpoint support. No error handling (cascading failure). Data passed via ad-hoc methods. |
| 1/5 | No pipeline capability. All steps executed in a single block with no stage separation. No resumability. |
