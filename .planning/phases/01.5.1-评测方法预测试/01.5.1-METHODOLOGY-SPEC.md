## 1. 概述

本规范定义了Phase 2全量评测使用的评分方法论、流程和输出格式。这份规范在Phase 01.5.1预测试（10个样本）中经过验证和迭代。

### 核心变更（对比旧体系）

| 项目 | 旧系统 | 新系统 |
|------|--------|--------|
| 功能维度 | 8个子维度各5分（40%） | 专业深度评分1-5（只评1-2个最佳功能） |
| 覆盖度维度 | 4个子维度各5分（30%） | 描述性信息（不评分） |
| 集成度维度 | 5个子维度各5分（30%） | 保持（评分+描述） |
| 决策驱动 | 加权总分100分 | Tier 1 深度评分（1-5） |
| 安全 | 独立否决 | 独立否决（同旧系统） |

### 两阶段评估架构

```
                     ┌──────────────────────────────┐
第一阶段 Tier 1:     │ 专业深度评分（1-5）           │
专业质量评级         │ 只评1-2个最佳功能             │ → 驱动决策：排除/候选/推荐
                     │ 决策阈值：<3排除 / 3-4候选 />4推荐 │
                     └──────────────────────────────┘
                                        │
                     ┌──────────────────────────────┐
第二阶段 Tier 2:     │ 补充信息（不评分）             │
组合决策输入         │ • 集成度评分（5个维度）        │ → 用于组合推荐
                     │ • 覆盖阶段描述                  │ → 人工确认最终组合
                     └──────────────────────────────┘
```

### Phase 2 上下文

- Phase 2 评测所有 39+ 个仓库
- 使用本规范验证过的评分方法论
- 输出：安全批准清单、全量评分卡、横向矩阵、组合推荐

## 2. 评估流程

### 2.1 整体流程

```
For each repository:
  1. Security check (4 items) → PASS → continue; FAIL → VETOED + record reason
  2. Function selection → pick 1-2 best functions per priority rules
  3. Score each selected function (1-5 per rubric) → DepthScore
  4. Apply threshold: <3 EXCLUDE, 3-4 CANDIDATE, >4 AUTO-RECOMMEND
  5. Record exclusion reason if EXCLUDE
  6. Score integration (5 dimensions, 1-5 each)
  7. Describe coverage stages descriptively
  8. Record all in TSV
```

### 2.2 安全检查（D-05）

使用与Phase 01.5.1相同的4项检查。从仓库文件中查找证据。任一FAIL → 否决。

| 检查项 | PASS条件 | 证据来源 |
|--------|----------|----------|
| 数据安全 | 不访问/存储敏感数据（PII、凭据、医疗记录） | package.json, .env, credential patterns |
| 权限范围 | 只申请最小必要权限 | manifest, 文件系统访问声明 |
| 网络请求 | 无可疑外部通信 | API调用, fetch, curl, 非标准端点 |
| 依赖来源 | 使用受信任的包注册源 | package.json, requirements.txt, Dockerfile |

### 2.3 功能选择（D-03）

**优先级规则（必须遵守此顺序）：**

1. 如果仓库README/描述中有自声明的专长领域 → 优先评分该功能
2. 如果声明了多个专长 → 评分其中证据最充分（代码质量、文档完整性）的1-2个
3. 如果没有明确专长 → 评分该skill提供最具体实现细节的2个功能

**选择原则：** 始终选择skill最擅长的功能，而非它"应该"做什么。

### 2.4 专业深度评分（D-03, D-06）

参照 01.5.1-SCORING-RUBRIC.md 中的标准对每个选定功能打分。

**评分规则：**
- 量表：1.0-5.0，0.5递增（共9级）
- 每个功能使用单独评分
- DepthScore = 选定功能分数的平均值，四舍五入到最近0.5
- 评分依据必须在评估报告中有具体证据引用

### 2.5 阈值映射（D-07, D-08, D-09）

| DepthScore | 分类 | 操作 |
|------------|------|------|
| < 3.0 | EXCLUDE | 必须附加排除理由（D-10） |
| 3.0 - 4.0 （含端点） | CANDIDATE | 自动分类，进入候选待选 |
| > 4.0 | AUTO-RECOMMEND | 自动进入组合推荐 |

边界定义：3.0 = CANDIDATE（下界包含），4.0 = AUTO-RECOMMEND（上界包含）。

### 2.6 排除理由（D-10, D-13）

格式：`[检查项名称] | [证据引用] | [为何重要]`

三个字段用管道符分隔。参考 01.5.1-EXCLUSION-REASON.md 中的完整模板。

### 2.7 集成度评分（Tier 2 — 保留评分）

使用与Phase 01.5.1相同的5个维度，每个1-5：

| 维度 | 1分含义 | 5分含义 |
|------|---------|---------|
| 集成难度 | 安装极其复杂 | 一键安装/复制即用 |
| MCP兼容 | 不支持MCP | 完整MCP协议支持 |
| 冲突风险 | 大量功能重叠 | 独特功能，无重叠 |
| 维护成本 | 已废弃/无更新 | 活跃维护，频繁更新 |
| 上下文依赖 | 高度依赖长上下文 | 自包含，低消耗 |

### 2.8 覆盖度描述（Tier 2 — 描述性，不评分）

记录两个字段：
- **Coverage_Stages:** 覆盖的科研阶段（文献/分析/写作/投稿），逗号分隔
- **Coverage_Description:** 2-3句话描述该skill覆盖的科研阶段和领域范围

## 3. 评分模板

### 3.1 TSV列定义

所有Phase 2评测数据使用以下22列表格格式：

| 列号 | 列名 | 类型 | 必需 | 说明 |
|------|------|------|------|------|
| 1 | Repo | text | 是 | 仓库名称 |
| 2 | Domain | text | 是 | 功能领域（如文献检索、数据分析） |
| 3 | 数据安全 | text | 是 | PASS或FAIL |
| 4 | 权限范围 | text | 是 | PASS或FAIL |
| 5 | 网络请求 | text | 是 | PASS或FAIL |
| 6 | 依赖来源 | text | 是 | PASS或FAIL |
| 7 | SecurityVerdict | text | 是 | APPROVED（全部PASS）或VETOED（任一FAIL） |
| 8 | Function1_Name | text | 是 | 选定的第一个功能名称 |
| 9 | Function1_Score(1-5) | number | 是 | 1.0-5.0，0.5步进 |
| 10 | Function2_Name | text | 否 | 选定的第二个功能名称（可选） |
| 11 | Function2_Score(1-5) | number | 否 | 1.0-5.0，0.5步进（可选） |
| 12 | DepthScore(1-5) | number | 是 | 平均分，四舍五入到最近0.5 |
| 13 | Threshold | text | 是 | EXCLUDE / CANDIDATE / AUTO-RECOMMEND |
| 14 | ExclusionReason | text | 条件 | 仅当Threshold=EXCLUDE时必需 |
| 15 | Integration_集成难度(1-5) | number | 是 | 1-5 |
| 16 | Integration_MCP兼容(1-5) | number | 是 | 1-5 |
| 17 | Integration_冲突风险(1-5) | number | 是 | 1-5 |
| 18 | Integration_维护成本(1-5) | number | 是 | 1-5 |
| 19 | Integration_上下文依赖(1-5) | number | 是 | 1-5 |
| 20 | Integration_小计 | number | 是 | 5个分数之和（5-25） |
| 21 | Coverage_Stages | text | 是 | 逗号分隔的阶段列表 |
| 22 | Coverage_Description | text | 是 | 2-3句描述 |

## 4. 重要提示与反模式

### 必须避免的反模式

1. **评分无关维度：** 不要给skill不相关的功能打分。一个文献检索工具的DepthScore不应因"代码生成能力弱"而扣分。
2. **将质量与兼容性混为一谈：** Tier 1（深度评分）和Tier 2（描述性信息）服务于不同的决策。不要因为集成困难而降低深度评分。
3. **过度定义评分标准：** 每个功能类型使用2-3个诊断性问题，不要使用10项检查清单。
4. **流水线与单功能skill的直接比较：** 流水线skill（如academic-pipeline）应作为流程编排者评分，不要与单功能skill直接比较分数。

### 预测试中发现的常见问题

| Issue | Severity | Impact | Recommended Fix |
|-------|----------|--------|-----------------|
| ARIS Literature Search score may be under-assessed if the skill's actual implementation has stronger search capability than described | MEDIUM | May affect Phase 2 ranking if not corrected | Phase 2 evaluators should examine ARIS's actual search implementation before assigning 3.0. If systematic search features exist, upgrade to 3.5 or 4.0 |
| academic-writing-skills score may be conservative — paper-audit multi-reviewer system suggests higher capability | MEDIUM | May affect Phase 2 ranking | Phase 2 should verify actual LaTeX template count and journal-specific support. If 3+ journal templates confirmed, upgrade to 4.0 |
| 0 EXCLUDE items in pre-test sample — threshold boundary cannot be validated | MEDIUM | Phase 2 will need deliberate low-quality sampling | Phase 2 sampling strategy should include 2-3 intentionally low-capability repos to test the EXCLUDE threshold |
| academic-pipeline scored conservatively due to documentation scarcity | LOW | May undervalue a capable pipeline | If more documentation becomes available, re-score Workflow Orchestration — current 4.0 may be conservative |
| No repo achieved 5.0 in any function — indicates either conservative rubric calibration or genuinely no "best-in-class" in this sample | LOW | May affect granularity of ranking within AUTO-RECOMMEND tier | Monitor Phase 2 results. If many repos cluster at 4.0, consider whether 4/5 criteria need calibration |
| Tie-breaking by alphabetical order is arbitrary for Phase 2 combination decisions | LOW | May lead to arbitrary ordering within AUTO-RECOMMEND tier | Phase 2 should use integration scores and coverage descriptions (Tier 2) to differentiate within tiers, not alphabetic ordering |
| All repos scored as APPROVED — no security veto cases to test exclusion workflow | LOW | Exclusion workflow (D-10/D-13) not validated | Phase 2 should include at least 1 repo with known security issues to test the veto + reason workflow |

## 5. 批准工作流（D-11, D-12）

### 自动化步骤（Claude执行）

1. 安全检查 → PASS/FAIL判定
2. 功能选择 → 按优先级规则
3. 各功能评分 → 按评分标准
4. DepthScore计算 → 平均后四舍五入
5. 阈值映射 → 排除/候选/推荐
6. 排除理由生成 → 结构化格式
7. 集成度评分 → 5个维度
8. 覆盖度描述 → 阶段+描述
9. 所有数据写入TSV

### 人工复核步骤（用户执行）

1. 排除项复核 → 确认排除理由合理（D-13）
2. 最终组合确认 → 选择最终组合方案（D-12）

### 工作流图示

```
[自动] 安全检查 → [自动] 功能评分 → [自动] 分类 → [自动] TSV记录
                                                       │
                                            [人工] 排除项复核 ←── 排除项
                                                       │
                                            [自动] 组合推荐生成
                                                       │
                                            [人工] 最终组合确认
```

## 7. Phase 2 执行建议

基于预测试发现，Phase 2执行应遵循以下建议：

1. **Rubric verification for borderline cases:** 在Phase 2评分前，验证ARIS文献检索实现和academic-writing-skills模板支持。这两个仓库当前分数最具不确定性。
2. **EXCLUDE category testing:** Phase 2评估时选择2-3个刻意选择的低能力仓库（如仅有README的单提示skill或明显废弃的仓库），以验证<3.0阈值产生可操作的排除。
3. **5/5 criteria calibration:** 如果Phase 2显示很多仓库评分为4.0但无人达到5.0，考虑是否需要将4/5标准拆分为4.0和4.5/5.0层级。
4. **Pipeline scoring guidance:** 对于流水线/编排器仓库，要求最低文档：阶段数量、检查点位置、错误处理方案和数据流schema。否则保守评分3.5/5。
5. **Security veto workflow testing:** Phase 2应至少包含1个已知安全问题的仓库，以验证排除工作流（D-10/D-13: 否决+结构化理由+人工复核）。
6. **Integration score role:** 集成分数（从旧评估保留）应作为AUTO-RECOMMEND层内的主要区分因子。建议将集成分数作为DepthScore后的二次排序键。
7. **Coverage descriptions for combination:** Tier 2覆盖描述应用于人工组合决策（D-12: 人工确认最终组合方案）。
