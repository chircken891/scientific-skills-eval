## 1. 概述

本规范定义了Phase 2全量评测使用的评分方法论、流程和输出格式。这份规范在Phase 01.5.1预测试（10个样本）中经过验证和迭代。

### 核心变更（对比旧体系）

| 项目 | 旧系统 | 新系统 |
|------|--------|--------|
| 功能维度 | 8个子维度各5分（40%） | 专业深度评分1-5（只评1-2个最佳功能） |

### 两阶段评估架构

```
                     ┌──────────────────────────────┐
第一阶段 Tier 1:     │ 专业深度评分（1-5）           │
专业质量评级         │ 只评1-2个最佳功能             │ → 驱动决策：排除/候选/推荐
                     │ 决策阈值：<3排除 / 3-4候选 />4推荐 │
                     └──────────────────────────────┘

                     ┌──────────────────────────────┐
第二阶段 Tier 2:     │ 补充信息（不评分）             │

                     │ • 覆盖阶段描述                  │ → 人工确认最终组合
                     └──────────────────────────────┘
```

## 2. 评估流程

### 2.1 整体流程

```
For each repository:
  1. Security check (4 items) → PASS → continue; FAIL → VETOED + record reason
  2. Function selection → pick 1-2 best functions per priority rules
  3. Score each selected function (1-5 per rubric) → DepthScore
  4. Apply threshold: <3 EXCLUDE, 3-4 CANDIDATE, >4 AUTO-RECOMMEND
  5. Record exclusion reason if EXCLUDE
  6. Score integration (5 dimensions, 1-5 each)
  7. Describe coverage stages descriptively
  8. Record all in TSV
```

### 2.2 安全检查（D-05）

使用与Phase 01.5.1相同的4项检查。从仓库文件中查找证据。任一FAIL → 否决。

| 检查项 | PASS条件 | 证据来源 |
|--------|----------|----------|
| 数据安全 | 不访问/存储敏感数据（PII、凭据、医疗记录） | package.json, .env, credential patterns |
| 权限范围 | 只申请最小必要权限 | manifest, 文件系统访问声明 |
| 网络请求 | 无可疑外部通信 | API调用, fetch, curl, 非标准端点 |
| 依赖来源 | 使用受信任的包注册源 | package.json, requirements.txt, Dockerfile |

### 2.3 功能选择（D-03）

**优先级规则（必须遵守此顺序）：**
1. 如果仓库README/描述中有自声明的专长领域 → 优先评分该功能
2. 如果声明了多个专长 → 评分其中证据最充分（代码质量、文档完整性）的1-2个
3. 如果没有明确专长 → 评分该skill提供最具体实现细节的2个功能

**选择原则：** 始终选择skill最擅长的功能，而非它"应该"做什么。

### 2.4 专业深度评分（D-03, D-06）

参照 01.5.1-SCORING-RUBRIC.md 中的标准对每个选定功能打分。

**评分规则：**
- 量表：1.0-5.0，0.5递增（共9级）
- 每个功能使用单独评分
- DepthScore = 选定功能分数的平均值，四舍五入到最近0.5
- 评分依据必须在评估报告中有具体证据引用

### 2.5 阈值映射（D-07, D-08, D-09）

| DepthScore | 分类 | 操作 |
|------------|------|------|
| < 3.0 | EXCLUDE | 必须附加排除理由（D-10） |
| 3.0 - 4.0 （含端点） | CANDIDATE | 自动分类，进入候选待选 |
| > 4.0 | AUTO-RECOMMEND | 自动进入组合推荐 |

边界定义：3.0 = CANDIDATE（下界包含），4.0 = AUTO-RECOMMEND（上界包含）。

### 2.6 排除理由（D-10, D-13）

格式：`[检查项名称] | [证据引用] | [为何重要]`

三个字段用管道符分隔。参考 01.5.1-EXCLUSION-REASON.md 中的完整模板。

### 2.7 集成度评分（Tier 2 — 保留评分）

使用与Phase 01.5.1相同的5个维度，每个1-5：

| 维度 | 1分含义 | 5分含义 |
|------|---------|---------|
| 集成难度 | 安装极其复杂 | 一键安装/复制即用 |
| MCP兼容 | 不支持MCP | 完整MCP协议支持 |
| 冲突风险 | 大量功能重叠 | 独特功能，无重叠 |
| 维护成本 | 已废弃/无更新 | 活跃维护，频繁更新 |
| 上下文依赖 | 高度依赖长上下文 | 自包含，低消耗 |

### 2.8 覆盖度描述（Tier 2 — 描述性，不评分）

记录两个字段：
- **Coverage_Stages:** 覆盖的科研阶段（文献/分析/写作/投稿），逗号分隔
- **Coverage_Description:** 2-3句话描述该skill覆盖的科研阶段和领域范围

## 4. 重要提示与反模式

### 必须避免的反模式

1. **评分无关维度：** 不要给skill不相关的功能打分。一个文献检索工具的DepthScore不应因"代码生成能力弱"而扣分。
2. **将质量与兼容性混为一谈：** Tier 1（深度评分）和Tier 2（描述性信息）服务于不同的决策。不要因为集成困难而降低深度评分。
3. **过度定义评分标准：** 每个功能类型使用2-3个诊断性问题，不要使用10项检查清单。
4. **流水线与单功能skill的直接比较：** 流水线skill（如academic-pipeline）应作为流程编排者评分，不要与单功能skill直接比较分数。
